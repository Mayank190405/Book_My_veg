Invoice Funding Platform – System Architecture

This system consists of two independent applications: Sandbox Infrastructure Platform and Service Provider Funding Platform.

1️⃣ Sandbox Infrastructure Platform

The Sandbox simulates real government and financial infrastructure systems. It behaves like external authorities such as UIDAI, GSTN, Credit Bureau, and Core Banking.

Purpose

  • Simulate identity verification (Aadhaar)
  • Simulate PAN & credit bureau scoring
  • Simulate GST registry and invoice data
  • Simulate banking system with ledger & transfers
  • Expose secure, government-style APIs

Core Components

  • Identity Registry – Stores Aadhaar & PAN mappings
  • Credit Bureau Module – Returns credit scores
  • GST Registry – Stores GSTIN & invoice history
  • Core Banking System – Handles accounts & transfers
  • NACH Module – Mandate creation & approval
  • API Gateway – Authentication, request tracking, structured responses

What It Does NOT Do

  • Does not calculate loan eligibility
  • Does not approve funding
  • Does not manage loan lifecycle
  • Does not own risk engine
User → Service Provider App → Sandbox APIs Sandbox verifies identity, credit, GST & processes fund transfers.

2️⃣ Service Provider Funding Platform

This is the actual fintech application that provides invoice funding to MSMEs. It owns all business logic and loan decision-making.

Purpose

  • User registration & authentication
  • Business onboarding
  • Invoice discovery (from Sandbox GST)
  • Risk scoring & eligibility calculation
  • Offer generation
  • Loan lifecycle management
  • Trust score & exposure tracking

Core Modules

  • User Management – Register/Login
  • Business Profile – GST & Bank linking
  • Invoice Engine – Fetch unpaid invoices
  • Risk Engine – Calculate funding percentage
  • Offer Engine – Generate funding offer
  • Loan State Machine – Track loan lifecycle
  • Trust Score System – Adjust funding eligibility

Loan Lifecycle

CREATED → UNDER_REVIEW → APPROVED → DISBURSED → ACTIVE → CLOSED / DEFAULTED

Interaction with Sandbox

  • Calls Aadhaar verification API
  • Calls Credit Bureau API for score
  • Calls GST API to fetch invoices
  • Calls Banking API for fund transfer
  • Calls Banking API for repayment processing

3️⃣ End-to-End System Flow

User ↓ Service Provider (Risk + Loan Logic) ↓ Sandbox Infrastructure (Verification + Banking)

Responsibility Separation

  • Sandbox = Authority + Financial Infrastructure
  • Service Provider = Fintech Brain + Loan Decision Engine

The Service Provider never directly modifies sandbox data. All communication happens via secured APIs.

4️⃣ Architectural Advantage

  • Clear separation of responsibilities
  • Replaceable sandbox with real government APIs in future
  • Scalable microservice-friendly design
  • Production-grade simulation environment
